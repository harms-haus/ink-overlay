## Runtime Safety & Environment Detection

The framework is designed to be imported anywhere — in a Node REPL, under Bun, inside a CI test runner, or even during a server-side render pass that never touches a terminal — without crashing or corrupting process state. This document explains the guard mechanisms that make that possible: the lazy, side-effect-free detection functions in `src/runtime.ts`, how the host in `src/host.tsx` uses them to gate raw mode and focus management, and the concrete caveats (Bun, non-TTY, CI, flicker) that follow from terminal and runtime realities. Every claim here traces back to code in those files plus the input dispatcher in `src/input-dispatcher.tsx`; for the broader system architecture, see [Architecture](./architecture.md), and for the Bun input limitation in depth, see [Platform Limitations](./limitations.md).

## Runtime Detection Functions

All environment detection lives in `src/runtime.ts` as standalone functions — not module-level constants evaluated at import time. This is a deliberate design choice: detection runs lazily, only when a function is called, which means the module's mere presence in a bundle has zero side effects. There are no top-level assignments, no IIFE blocks, and no eager reads of `process.env` or `process.stdout`. The file exports three detection functions, one snapshot helper, one warning emitter, and a `RuntimeInfo` type.

### isBun()

`isBun()` checks whether `globalThis.Bun` is defined. Bun sets this global on startup; Node does not. The check is a single property access — `globalThis.Bun !== undefined` — wrapped so it never throws even if the global object is in an unusual state. The return type is a plain boolean. This function is used in two places: `getRuntimeInfo()` populates its `bun` field from it, and `warnBunInput()` gates the one-time warning on it. Neither call site invokes `isBun()` at module load.

The reason for a dedicated function rather than inlining the check is twofold. First, it centralizes the detection so that if Bun ever changes its global identifier (or if a new runtime emerges with a similar pattern), there is a single place to update. Second, it makes the intent self-documenting at call sites — `if (isBun())` reads more clearly than `if (globalThis.Bun !== undefined)`, and the JSDoc on the function serves as inline documentation for anyone reading the host code. The function is exported from the package barrel, so consumers can also use it in their own environment-gated logic.

### isNonInteractive()

`isNonInteractive()` mirrors Ink's own interactive detection. It returns true when `process.stdout` lacks an `isTTY` property (meaning output is piped, redirected, or not connected to a terminal) **or** when `process.env.CI` is set to any truthy value. The function uses optional chaining on `process.stdout` (`process.stdout?.isTTY`) so it does not throw if stdout is somehow absent. The CI check is important because many CI systems allocate a pseudo-TTY for stdout but are still not interactive — tests and build pipelines should never attempt to enter raw mode or wait for keypresses. The function short-circuits on the TTY check first, returning early when stdout is not a TTY, so the CI environment variable is only consulted when stdout already passes the TTY test. This mirrors the precedence Ink itself uses.

### isRawModeSupported()

`isRawModeSupported()` returns `Boolean(process.stdin?.isTTY)`. This is a separate check from the stdout TTY test in `isNonInteractive()` because stdin and stdout can have different TTY status. For example, the shell command `echo foo | node app.js` gives stdout a TTY (in most setups) but replaces stdin with a pipe. Raw mode is a property of the stdin file descriptor specifically — it tells the terminal driver to pass each byte through immediately without line buffering or signal generation — so it must be tested on stdin, not stdout. The optional chaining on `process.stdin` prevents a crash if stdin is undefined in an exotic environment.

### getRuntimeInfo() and the RuntimeInfo Type

`getRuntimeInfo()` returns a snapshot object with three boolean fields — `bun`, `interactive`, and `rawModeSupported` — by calling the three detection functions described above. The `bun` field comes from `isBun()`. The `interactive` field is the logical negation of `isNonInteractive()`: it is true when the session appears to be genuinely interactive (stdout is a TTY and CI is not set). The `rawModeSupported` field comes from `isRawModeSupported()`. The `RuntimeInfo` type is internal and not re-exported from the package barrel; consumers who want to annotate the return value should use `ReturnType<typeof getRuntimeInfo>` from the package import. The demo's [Runtime & Environments scene](../../demo/README.md) renders a live readout of `getRuntimeInfo()` and the scalar helpers. Because each field is computed at call time, the snapshot reflects the environment at the moment of the call, not at import time — useful if the process's TTY status could theoretically change, though in practice it does not.

### warnBunInput()

`warnBunInput()` is a lazy, one-shot warning emitter. It is never called at module load — only when a capturing layer first mounts inside the host (see the Bun Caveats section below for the call site). A module-level boolean flag `bunWarnEmitted` guards against duplicate warnings across the process lifetime: once the flag is set to true, subsequent calls return immediately. If the runtime is not Bun, the function returns before doing anything, without touching the flag. The warning message identifies the library, names the specific upstream issue (bun#6862), explains what will and will not work, and recommends Node for interactive features.

The one-shot latch pattern is important for two reasons. First, without it, the warning would fire every time a capturing layer mounts — potentially dozens of times in a session as modals open and close — flooding the console with identical messages. Second, the latch is module-scoped rather than instance-scoped: even if an application mounts and unmounts multiple `OverlayHost` instances over its lifetime, the warning fires at most once for the entire process. This is the only mutable module-level state in `src/runtime.ts`, and it is a simple boolean that never touches process I/O, so it does not violate the no-side-effects-on-import invariant.

### Why Detection Is Function-Based, Not Constant-Based

It is natural to ask why these are functions rather than top-level constants — why not write `const IS_BUN = globalThis.Bun !== undefined;` and export that? The answer is the no-side-effects invariant combined with testability. A top-level constant is evaluated eagerly at import time: the moment the module is loaded, the constant is computed and frozen. In most real environments this produces the same result, but it means the detection is permanently baked in and cannot be re-evaluated. With function-based detection, the same module can report different results if the environment changes between calls (rare but possible in long-lived processes or test harnesses that manipulate global state). More importantly, the lazy evaluation means that importing the module in a context that will never call these functions — a server render pass, a build step — incurs zero cost and zero risk, because the detection code never executes at all.

## How the Host Gates Raw Mode and Focus

The runtime detection functions are passive — they report state. The active enforcement happens inside `OverlayHost` via a set of React effects that manage raw mode and Ink's focus manager based on the number of currently-capturing layers. Understanding this mechanism is essential to understanding why the framework is safe in non-interactive environments.

During each render, the host computes `capturingCount`: the number of layers in the sorted list that have `capture` set to true **and** are not in the process of exiting (the `exiting` flag is set during exit-transition animations). This count is the single source of truth for whether the terminal should be in raw mode and whether Ink's focus navigation should be disabled. A ref (`previousCapturingCountReference`) stores the count from the previous commit so the effect can detect transitions rather than acting on every render.

The raw-mode effect fires when `capturingCount` changes. If the count transitions from zero to positive, the effect calls `setRawMode(true)` — but only when `isRawModeSupported` (from `useStdin`) is true. If the count transitions from positive back to zero, it calls `setRawMode(false)`, again gated on `isRawModeSupported`. Both calls are wrapped in try/catch; a failure logs a warning but does not crash. A parallel focus-management effect calls `disableFocus()` on the 0-to-positive transition and `enableFocus()` on the positive-to-zero transition, trapping Tab/Shift+Tab navigation within the capturing layer.

A separate unmount-cleanup effect runs only on host teardown. It reads the ref (not the setup-time value) so it always sees the last-committed capturing count. If that count was positive — meaning raw mode was active when the host unmounted — the cleanup calls `setRawMode(false)` and `enableFocus()`. This is the mechanism that restores terminal state on clean exit, and it is the one that does not fire on abrupt process death (relevant under Bun, as discussed below).

The critical safety property is that every `setRawMode` call — in all three effects — is gated on `isRawModeSupported`. In a non-TTY or CI environment, that value is false, so raw mode is never touched. The count may rise and fall as layers open and close, but the effects are no-ops with respect to the terminal driver. This is what makes the framework safe to mount in any environment.

## SSR, Non-TTY, and CI Graceful Degradation

When `isNonInteractive()` returns true, the framework degrades gracefully rather than throwing or hanging. The key guard is in `OverlayHost`: the host obtains `isRawModeSupported` from Ink's `useStdin()` hook and only calls `setRawMode(true)` when that value is truthy. In a non-TTY or CI context, `useStdin().isRawModeSupported` is false, so raw mode is **never** enabled. The same guard wraps the `setRawMode(false)` call on the teardown path, so there is no asymmetry between setup and cleanup. Both calls are also wrapped in try/catch blocks that log a warning on failure, ensuring that even a surprising `setRawMode` throw does not crash the render cycle.

Rendering is decoupled from input entirely. `OverlayHost` renders its layers through `LayerRenderer` during the normal React render cycle — this path has no dependency on raw mode, keypress events, or TTY status. Overlays paint their final, fully-resolved frames to stdout regardless of whether anyone can type at a keyboard. The host's render output (the relative-positioned box, the absolute-positioned layer children, the backdrop fills) is produced purely from layout math and React reconciliation. No part of the render path calls `setRawMode`, reads from stdin, or waits for input events.

This decoupling matters for two scenarios. First, SSR or pre-render contexts: a snapshot of the overlay's static output can be captured without an interactive session, which is useful for generating documentation screenshots, validating layout in CI, or rendering to a string for log output. The rendered output is deterministic given the same props and terminal dimensions — there is no randomness or input-dependent branching in the paint path. Second, CI test environments using `ink-testing-library` or similar fake-stdout harnesses: tests assert on rendered frames via `lastFrame()`, not on live keyboard interaction, and the framework cooperates because rendering never blocks on input and never attempts to enter raw mode against the fake stream. The fake stdout provided by `ink-testing-library` reports `isTTY: false` (or lacks `isTTY` entirely), so the `isRawModeSupported` guard prevents any attempt to toggle raw mode on the mock.

The consequence is that keyboard input is non-functional in non-interactive contexts — no keypress events arrive, so `useInput` handlers never fire, focus traps cannot cycle, and Escape-based dismissal does not work. This is consistent with Ink's own `interactive: false` behavior, where Ink internally gates input handling on the same TTY and CI conditions. The framework does not attempt to paper over this limitation; it simply ensures that the rendering path continues to work so that tests and static output remain valid. The `captureDepth` signal and the input dispatcher continue to function internally — capturing layers still register on the LIFO stack and `useInputCaptureState()` still reflects the capture state — but no external keypresses arrive to trigger any of it.

## No Global Side Effects on Import

A critical invariant of `src/runtime.ts`, stated explicitly in its header comment, is that importing the module produces **no global side effects**. There are no top-level calls to `process.stdin.setRawMode`, no `process.stdin.on('data', …)` listener registration, no mutation of `process.env`, and no monkey-patching of global objects. Every side effect — entering raw mode, disabling Ink's focus manager, emitting the Bun warning — is deferred to a function call or a React effect that runs only when an `OverlayHost` actually mounts and a capturing layer transitions the capture count above zero.

This choice was made for two reasons. First, SSR and isomorphic rendering: a server bundle that imports the package (perhaps because the same component file is shared between client and server code) must not grab stdin or alter terminal state on the server. If the module touched `process.stdin` at import time, merely loading it would put the server process into raw mode or attach dangling listeners that would never be cleaned up, corrupting the server's own I/O. Second, test isolation: test runners like Jest and Vitest import many modules into the same long-lived process. A module that mutated global state on import would leak between test files, cause flaky or order-dependent failures, and make reliable teardown impossible. By keeping detection lazy and all mutations inside React's effect lifecycle — where setup and cleanup are paired — the framework guarantees that importing it is always safe and that every side effect has a well-defined cleanup counterpart that runs on unmount.

It is worth contrasting this with what a naive implementation might do. A common pattern in terminal libraries is to enter raw mode at the top level of the entry module, or to attach a global `keypress` listener at import time. That approach is simpler to write but makes the module impossible to use safely in any context other than a single-process interactive CLI. The framework's approach — deferring every side effect to the host's effect lifecycle, gating on `isRawModeSupported`, and keeping detection purely functional — costs a few extra lines of code but buys universal import safety. The only module-level mutable state in the entire runtime module is the `bunWarnEmitted` boolean, and that is a simple latch that never interacts with process I/O.

## Bun Caveats

Under Bun, rendering works normally — `OverlayHost`, `LayerRenderer`, backdrop overpaint, transitions, and all visual features produce correct output because they rely only on Ink's render pipeline, not on input. However, interactive input is non-functional due to [bun#6862](https://github.com/oven-sh/bun/issues/6862): Bun does not call `process.stdin.resume()`, so the readable stream never enters flowing mode and no keypress events are emitted to Ink's internal input handler. This means `useInput` callbacks never fire, focus traps cannot intercept Tab or Shift+Tab, and keyboard dismissal (Escape, click-away via `onBackdropInput`) is inert. Overlays will render but not respond to the keyboard.

The InputDispatcher component (`src/input-dispatcher.tsx`) calls `warnBunInput()` from a useEffect that fires the first time its `captureDepth` counter transitions to 1 or higher — i.e., when the first capturing overlay or focus trap registers. The effect is guarded by a `bunWarnFired` ref so it fires at most once per process. The warning is intentionally deferred to this point rather than emitted on import, because non-interactive Bun processes (build scripts, test snapshots, pre-render passes) should not be penalized with a spurious console warning when they never attempt interactive input. The warning fires only when the framework detects that interactive input is actually being requested.

Raw mode restoration has a second Bun-specific caveat: [bun#12918](https://github.com/oven-sh/bun/issues/12918) means raw mode may not be restored correctly on abrupt process exit under Bun. The framework restores raw mode on **clean unmount** of `OverlayHost` — the cleanup effect in `src/host.tsx` reads the last-committed capturing count from a ref (not the value at setup time, which would be stale because the effect's dependencies are stable and setup runs only once) and, if that count is positive, calls `setRawMode(false)` and calls `enableFocus()`. But if the process exits abruptly (for example, via an uncaught SIGINT or a hard crash), this cleanup effect never runs, and the terminal may be left in raw mode.

Under Bun, consumers who need interactive features are strongly advised to use Node instead. The rendering path is fully functional under Bun, so overlays that are purely display-oriented (toasts, static information panels, auto-dismissing notifications driven by timers rather than keyboard input) work correctly. Only overlays that depend on capturing keyboard events — modals with `capture`, command palettes, focus-trapped dialogs — are affected. Those running under Bun who accept the input limitation should register a `process.on('SIGINT', …)` handler as a safety net to disable raw mode and restore focus on exit, compensating for the cleanup effect that will not fire on abrupt termination. For the full discussion of this limitation's root cause and whether it is addressable upstream in Bun, see [Platform Limitations](./limitations.md).

## Flicker

Ink performs a full-screen clear under two conditions that overlap with overlay rendering. First, when a rendered frame's height is greater than or equal to `stdout.rows`, Ink clears the entire screen before painting — this is its mechanism for handling full-height output without leaving stale characters from the previous frame. Second, when the terminal **width decreases** between frames (as detected by Ink's resize handler, which compares the current column count against the previously observed width), Ink clears and repaints from scratch to avoid corrupting the layout.

The practical implication for overlays is that a layer whose content reaches or exceeds the terminal's row count will trigger the full clear on every frame, producing visible flicker as the screen blanks and redraws. The host renders inside a `<Box>` whose height is set to `terminalRows` (sourced from Ink's `useWindowSize` hook), so a layer that fills that full height is right at the threshold. The recommendation is to keep overlay content within `rows - 1` to stay safely under the clear trigger. This is a soft guideline, not a hard API constraint — the framework does not clip or reject content that exceeds the limit — but respecting it avoids the flicker entirely. Height-based clears are inherent to Ink's output model and cannot be disabled or worked around by the framework.

Width-based clears are triggered by terminal resize events where the new column count is smaller than the previous one. Since overlays often react to size changes — re-anchoring to a new viewport, re-flowing wrapped text, recomputing popover placement — resize-driven flicker is common in practice. The mitigation is to pass `incrementalRendering: true` in the host application's `render()` call (the Ink `render` function from the `ink` package, not anything in this framework). Incremental rendering tells Ink to write only the diff between frames instead of clearing and repainting the whole screen, which substantially reduces flicker on resize and on large frame-to-frame changes. This is an Ink-level option; the framework does not set it automatically because it must be configured at the `render()` call site, which the consumer owns and which exists outside the framework's component tree.

A related but distinct flicker source is animation transitions, which produce rapid successive frames during enter and exit animations. Because each animation step is a separate React render producing a new frame, a multi-step transition can generate several full-screen repaints in quick succession. Keeping transition step counts modest and durations reasonable (the built-in transitions use 2–3 steps at 80ms intervals) limits this. For more on the animation system and its frame model, see [Animation](./animation.md). The `rows - 1` height guideline and `incrementalRendering` recommendation together address the majority of real-world flicker scenarios.
